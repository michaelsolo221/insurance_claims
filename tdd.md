## Agent Design

### Architecture

**Modality:** Phone / voice, via Google Telephony Platform (GTP) attached directly to the CX Agent Studio (CES) app. Because this is a voice agent, the model is `gemini-3.1-flash-live` per platform guidance (requirements-interview.md §2). Stakeholder confirmed default platform speaking rate and voice persona are acceptable -- no custom `speakingRate` override needed (§2). Language: English only at launch, no multilingual support (§2, §10).

**Persona (new, §12):** The agent has a named persona, **"Amanda."** Every call must open with a fixed, scripted greeting: *"Good day, you've reached CGU express claim lodgements, my name is Amanda, how can I help you?"* (adapted only for grammatical fit within the turn). Per `gecx-design-guide.md` -> "Callback Patterns for Deterministic Behavior" ("Deterministic greeting: `before_model_callback` intercepts session start and returns static greeting"), this fixed opening line is exactly the kind of behavior that should be callback-enforced rather than left to LLM improvisation on the first turn -- see Callbacks below. This does not create a second persona/CUJ that would justify a sub-agent; it's a naming/voice detail layered onto the single existing conversational flow.

**Region and GCP project (revised, §12 supersedes §3):** CES `us` multi-region. No Australia region exists on CES; stakeholder has explicit internal approval to accept `us`-region hosting for this workload, including production (§3, §10). **§12 reverses the prior project decision:** the agent is to be built in the **existing** GCP project `claims-lodgement-20260628`, under gcloud account `michael.solo@mcosolutions.com.au` -- NOT a fresh project. The earlier "stale/unrelated, create fresh" framing (§3) no longer applies and is superseded; this project is the confirmed target for both prototype and production use.

**Proposed hierarchy: single agent, no sub-agents.**

- **Root Agent** (`root_agent`, no `childAgents`) -- handles the entire conversation: the fixed greeting, FNOL (First Notice of Loss) data capture, the emergency guardrail, and the out-of-scope decline-and-redirect behavior.

Justification for single-agent (per `references/gecx-design-guide.md` -> "Start Simple" / "Multi-Agent" criteria): the source describes exactly **one** in-scope conversational journey -- lodging a new claim (§4, "Single in-scope intent"). The other behaviors the agent must handle (greeting, emergency override, out-of-scope decline) are not separate CUJs requiring their own persona or context -- they are a fixed opening line and short, deterministic redirects that terminate the call rather than branching into a different multi-turn flow. The design guide's multi-agent trigger conditions ("if the role/persona are meaningfully different," "multiple disjoint CUJs," "highly complex CUJ that breaks into sequential steps") do not apply here: there is one persona, one CUJ, and no sub-flow complex enough to need isolation. Sources do not describe or imply any second capability that would justify a sub-agent -- do not add one.

**Field-collection implementation: Slot-Filling DAG Framework (confirmed).** Per §11 Round 2 resolution: "confirmed -- use the platform's Slot-Filling Framework (deterministic, code/tool-visibility driven state), not a simple instruction-driven taskflow. This applies to all fields listed in section 6." This drives the Tools, Variables, and Callbacks sections below. Per `gecx-design-guide.md` -> "Slot Filling Framework": this pattern is intended for agents whose primary job is "collecting structured data to fire backend operations," which matches this agent's FNOL-capture-then-submit design. The framework splits responsibility three ways, all implemented in one callback file per the design guide's convention:

1. **`_get_config()`** -- agent-specific slot definitions for the 9 FNOL fields (§6), the single backend task (`submit_claim_notification`, §7/§12), and the confirmed per-slot retry policy (see below). This is where the retry/escalation policy lives, per the design guide's principle "Lean on the orchestrator -- if a constraint is enforced by code (validation, tool visibility, retry logic), don't duplicate it in prompts or tool docstrings."
2. **`_run_slot_filling(config, sm)`** -- the CES-agnostic orchestrator. On each turn, computes which setter tools are valid to show (via `hide_tool()`), tracks per-slot attempt counts, decides when `submit_claim_notification` becomes callable (all required slots filled + readback confirmed), and decides when a required slot has exhausted its retries.
3. **`before_model_callback()`** -- the thin CES adapter (~20 lines per the design guide) that applies the orchestrator's output to the actual turn: sets tool visibility, injects `_system_message`, handles preemption, and (new, §12) speaks the fixed greeting on the first turn.

**Retry / escalation policy (confirmed, §11):** "retry up to 2 additional times (3 total attempts) if the caller can't or won't provide a required field, then proceed without it, marking that field as 'not captured' in the submitted email, rather than blocking the call indefinitely." Per the design guide's Slot Filling Framework section, this is orchestrator-owned logic (`_get_config()` per-slot retry limit + `_run_slot_filling()` attempt counting), not an instruction. This policy applies only to **required** slots (caller full name, callback phone number, incident date/time, incident location, incident description, injury flag, hazard flag); the two **optional** slots (policy number, preferred callback window, both explicitly optional per §6) are not subject to retry escalation -- a single "don't know" ends collection for that slot immediately.

**Emergency + submission-outcome determinism (confirmed, §11; submission outcome revised, §12):** The `flag_emergency` tool and `before_model_callback` (deterministic hang-up-and-call-000 redirect) proposed in the first TDD draft are stakeholder-confirmed as the correct way to satisfy the hard requirement in §5. The `after_model_callback` submission-outcome acknowledgment pattern is also confirmed, but **§12 changes what outcome it gates on** -- Firestore write success/failure, not Gmail send success/failure (see Tools, Callbacks, and Known Issues below).

**Priority ordering (confirmed, §11):** "the emergency override always takes priority if both could apply to the same utterance (not expected to co-occur in practice, but ruled on for completeness)." Concretely: the same `before_model_callback` that runs the slot-filling orchestrator checks `_emergency_flag` **first** and preempts (hang-up-and-call-000 message + `end_session`) before running `_run_slot_filling()` or evaluating any out-of-scope routing. See Routing Logic.

### Tools

| Tool Name | Type | Purpose | Source |
|---|---|---|---|
| `submit_claim_notification` | Python function | **Redesigned, §12.** The Slot-Filling Framework's backend task: fires once all required FNOL slots are filled and the caller has confirmed the readback. Two-step behavior: **(1) synchronous, fast** -- writes the full claim record (including any slots marked "not captured" per the retry-exhaustion policy) to a **Firestore** collection; this durable write is the only operation the live call waits on, and is the new source-of-truth / success criterion the tool returns to the agent. **(2) decoupled, async** -- once the Firestore write succeeds, enqueues a **Cloud Task** that will trigger a separate Cloud Function (see Infrastructure row below) to send the confirmation email via Gmail API to `michael@michael-lo.com` *after* the call has already ended. Always visible regardless of slot-filling state -- per the confirmed priority ordering (§11), an emergency must be flaggable at any point in the conversation, not gated behind slot-filling tool visibility. | §5 (hard requirement) + §11 ("confirmed -- build the proposed `flag_emergency` tool... as designed in the draft TDD"). |
| `end_session` | System (platform built-in) | Terminates the call after: (a) successful-submission acknowledgment (Firestore write succeeded, §12), (b) a failed-submission fallback message (Firestore write failed, §12), (c) an out-of-scope decline-and-redirect message, or (d) the emergency-redirect message. | Not named explicitly by the source, but required for any voice-modality agent to end a call; inferred from modality (§2), not fabricated business logic. |

**Error handling for `submit_claim_notification` (revised, §12):** per `gecx-design-guide.md` -> Error Handling Guidelines, on **Firestore write failure** the tool should return a structured result with an `agent_action` key giving the LLM a deterministic recovery instruction (apologize, take a callback number), rather than a generic "an error occurred" string. This directly implements §7/§12's requirement that the caller must never be left with no acknowledgement that submission failed -- but the trigger for this path is now Firestore, not Gmail. The Cloud Task enqueue step is fire-and-forget from the live call's perspective; if Cloud Task enqueue itself fails after a successful Firestore write, that is an infrastructure-level failure of the async email hand-off, not a caller-facing failure -- the record is already durable. Sources do not specify retry/alerting behavior for a failed Cloud Task enqueue or a failed Cloud Function email send; flagged as an open question (see Known Issues).

**No other tools are proposed.** Sources explicitly confirm there is no backend claims system, policy admin system, or API to integrate with (§7: "none, the agent will capture via email and send to mailbox.") -- Firestore and the async Cloud Function are new infrastructure introduced by §12's redesign, not an existing system being integrated with.

### Routing Logic

There is no multi-agent routing (single agent, see Architecture). "Routing" here means which response branch the root agent takes based on what the caller says, and both branches below are fully specified (no remaining ordering ambiguity):

- **Call open (new, §12):** every call begins with the fixed greeting -- *"Good day, you've reached CGU express claim lodgements, my name is Amanda, how can I help you?"* -- before any other routing branch is evaluated.
- **In-scope (FNOL lodging):** proceed with the Slot-Filling Framework data-capture flow described in Variables/Coverage Map. Source: §4, "Single in-scope intent: caller wants to lodge/report a new insurance claim (FNOL)."
- **Out-of-scope intent detected** (status check on an existing claim, a direct request to speak to a human, or any other general inquiry): the agent must decline and redirect using the **confirmed, verbatim script** (§11): *"This phone number is only for express claim lodgements; please call CGU general enquiries on 1800 248 224."* This exact wording (adapted only for grammatical fit within the turn) is required for **every** out-of-scope case in §4. No live transfer tool/child agent should be built for this path (§4: "It does NOT attempt a live transfer/handoff for these cases.").
- **Emergency override (highest priority, can interrupt at any point):** "If at any point the caller's language indicates an active emergency or danger in progress..., the agent must immediately stop normal FNOL intake and instruct the caller to hang up and call 000... This takes priority over completing the data-capture flow." (§5). This must be checked on every turn, not just at call start.
- **Priority ordering, emergency vs. out-of-scope (confirmed, §11):** "the emergency override always takes priority if both could apply to the same utterance (not expected to co-occur in practice, but ruled on for completeness)." Implementation: `before_model_callback` evaluates `_emergency_flag` before evaluating out-of-scope routing or running the slot-filling orchestrator, on every turn.
- **No auth-based routing.** The call is fully unauthenticated -- "no verification, we accept it as is, after its sent to the email, then we verify (offline)" (§8). There is no branch for authenticated vs. unauthenticated callers; every caller gets the same unauthenticated flow.

## Known Issues

Source-stated accepted tradeoffs (requirements-interview.md §10, still applicable):
- No Australia region exists on CES; `us` region accepted by stakeholder for both prototype and production use.
- No identity verification occurs on the call; accepted as a deliberate design choice -- verification is a downstream, offline, human step after the email/Firestore record is received.
- English only; no multilingual support this version.

Superseded / resolved items (no longer open, carried here for traceability only):
- **GCP project (§3, superseded by §12):** the prior draft called for a fresh GCP project, treating `claims-lodgement-20260628` as stale/unrelated. §12 reverses this -- the agent will be built in the existing project `claims-lodgement-20260628` (gcloud account `michael.solo@mcosolutions.com.au`). This is now a confirmed design input, not a tradeoff or open question.
